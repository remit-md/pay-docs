# Pay — x402 Details

Most of the time, `pay request <url>` handles everything. This
reference is for when you need to understand what's happening under
the hood or when something goes wrong.

## How `pay request` works

```
1. Agent runs: pay request https://api.example.com/data
2. CLI sends GET to the URL
3. If response is 200 → return response (not paywalled)
4. If response is 402 → CLI reads PAYMENT-REQUIRED header
5. CLI determines settlement mode (direct or tab)
6. For direct: CLI signs EIP-3009 authorization, retries with payment
7. For tab: CLI checks for open tab with provider
   - If none: auto-opens tab (10x per-call price, minimum $5)
   - If exists: charges existing tab
   CLI retries with tab charge proof
8. Provider verifies via facilitator → serves content
9. CLI returns the API response
```

Steps 4-8 are invisible. The agent just gets the response.

## Settlement modes

The provider declares which settlement mode to use in their 402
response:

### Direct settlement (`settlement: "direct"`)

- On-chain payment per request
- For calls priced at $1+
- Standard x402 — works with any x402-compatible agent SDK
- Higher latency (~2s for on-chain confirmation)

### Tab settlement (`settlement: "tab"`)

- Charges against a pre-funded tab
- For any price point, especially micropayments
- Pay-specific — requires the Pay CLI or SDK
- Low latency (~50ms, off-chain verification)
- Batched settlement (near-zero effective gas per charge)

The CLI handles both modes transparently based on the 402 response.

## Auto-tab-open behavior

When `pay request` hits a tab-settlement 402 and no tab exists with
that provider:

- CLI auto-opens a tab: 10x the per-call price, minimum $5
- Activation fee is deducted ($0.10 minimum)
- This happens without prompting

After auto-open, the charge proceeds and the response is returned.
The agent should note that a tab was opened and report it.

## Facilitator

The facilitator is server-side infrastructure. The 402 response does
NOT include the facilitator URL — per the x402 spec, it's purely a
server concern. The agent only sees the price, network, asset, and
provider address.

The flow:
1. Agent gets 402 with payment requirements (no facilitator info)
2. Agent signs a payment and retries with `PAYMENT-SIGNATURE`
3. Server (pay-gate) verifies with its facilitator internally
4. Server returns the response

The agent never calls the facilitator directly.

### The Pay facilitator

```
URL:     https://pay-skill.com/x402
Testnet: https://testnet.pay-skill.com/x402

Endpoints:
  POST /x402/verify     Verify payment (off-chain, instant)
  POST /x402/settle     Execute on-chain settlement
  GET  /x402/supported  List supported schemes/networks
```

### Non-Pay facilitators

Not all 402 responses use the Pay facilitator. The x402 ecosystem
includes other facilitators (e.g., from Coinbase).

If `pay request` encounters a 402 from a non-Pay provider and the
settlement flow is incompatible, it reports that a different x402
client may be needed.

## pay-gate

pay-gate is the provider-side reverse proxy that generates 402
responses and verifies payments. The agent never interacts with
pay-gate directly — it's transparent. The agent just sees a 402
from the API URL.

For provider setup docs: pay-skill.com/docs/gate

## Wire format

x402 V2 headers (the only version Pay supports):

| Header | Direction | Purpose |
|--------|-----------|---------|
| `PAYMENT-REQUIRED` | server �� agent | Base64 JSON payment requirements |
| `PAYMENT-SIGNATURE` | agent → server | Payment proof |
| `PAYMENT-RESPONSE` | server → agent | Settlement receipt |

V1 headers (`X-PAYMENT`, `X-PAYMENT-RESPONSE`) are not supported.

### PAYMENT-REQUIRED contents

```json
{
  "x402Version": 2,
  "resource": { "url": "/api/data", "description": "...", "mimeType": "..." },
  "accepts": [{
    "scheme": "exact",
    "network": "eip155:8453",
    "amount": "10000",
    "asset": "0x833589fCD6eDb6E08f4c7C32D4f71b54bdA02913",
    "payTo": "0xprovider...",
    "maxTimeoutSeconds": 60,
    "extra": { "settlement": "direct", "facilitator": "https://pay-skill.com/x402" }
  }],
  "extensions": {
    "bazaar": {
      "info": {
        "input": {
          "type": "http",
          "method": "GET",
          "queryParams": { "q": { "type": "string", "required": true } }
        },
        "output": { "type": "json" }
      },
      "schema": { "$schema": "https://json-schema.org/draft/2020-12/schema", "..." : "..." },
      "routeTemplate": "/api/data"
    }
  }
}
```

The `extra` object contains Pay extensions. `extra.settlement` declares
the settlement mode. `extra.facilitator` is the URL of the facilitator
that will verify and settle the payment. Standard x402 fields
(`scheme`, `network`, `amount`, `asset`, `payTo`, `maxTimeoutSeconds`)
are fully spec-conformant.

### Bazaar extension (`extensions.bazaar`)

When a provider configures an `info` block on a route, the 402
response includes `extensions.bazaar`:

| Field | Description |
|-------|-------------|
| `info` | Structured input/output description from the route config |
| `schema` | Auto-generated JSON Schema (draft 2020-12) for request validation |
| `routeTemplate` | URL template with named params (e.g. `/users/:id`), if configured |

Agents should read the info block to understand what parameters the
route accepts before paying. pay-gate validates requests against the
info schema after payment -- invalid requests get `400 Bad Request`
and the payment is not refunded.

Routes without an `info` block emit `"extensions": {}`.

### Pre-flight discovery

Agents can inspect a service's payment terms and input requirements
before triggering a 402:

- `GET /.well-known/x402` -- IETF draft standard descriptor (includes info blocks per endpoint)
- `GET /__pay/manifest` -- Pay-specific legacy alias
- DNS TXT at `_x402.<hostname>` -- points to the descriptor URL

The `.well-known/x402` endpoint is the recommended way for agents to
discover what parameters an API accepts before paying. Each endpoint
in the response includes its `info` block (if configured) and
`routeTemplate` (if applicable).

## `pay request` options

Mirrors curl:

```
pay request <url>                    GET request (default)
  -X POST                           HTTP method
  -H "Authorization: Bearer ..."    Headers (repeatable)
  -d '{"query": "..."}'             Request body
  -d @file.json                     Body from file
  -o output.json                    Write response to file
  -v                                Show request/response headers
  -s                                Body only (no status line)
  --no-pay                          Skip payment handling (debug)
```
